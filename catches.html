What MDLC catches.

Not a feature list — anyone can claim "secure." Every item below is a real finding from the adversarial gates on the eight reference builds, and each one links to the SECURITY-AUDIT.md it came from. Caught during the build, then auto-fixed, runtime-verified, or honestly dispositioned.

02Authentication

Including a critical account-takeover that the build's own report had already marked "done."

CRITICAL — account takeover via re-registration

POST /auth/register with an existing email returned the victim's record and issued a valid session cookie scoped to their account. Caught, auto-remediated (existing-email now returns a shape-identical 201 with no cookie and no data), and surfaced to a human anyway, per the kernel's critical-finding rule.

Unknown email and wrong password return the same 401; login runs a real password-hash verify against a constant DUMMY_HASH for unknown emails so response timing can't reveal which accounts exist.

The limiter runs before any argon2/bcrypt/scrypt verify — so a flood of login attempts can't be amplified into a CPU-exhaustion DoS through the hashing step.

03Claimed-but-inert controls

The signature catch. The build's own spec and report said a control was "Applied." The adversarial pass grepped file:line and proved it wasn't — then fixed it or escalated it to a human rather than letting the lie ship.

Production boots with publicly-known dev JWT keys (HIGH → auth bypass)

Config fell back to dev-access-secret-change-me and compose set NODE_ENV=production with those same defaults — meaning anyone could forge access/refresh tokens and bypass auth entirely. Fixed: assertProductionSecrets() aborts boot when secrets are unset, dev-default, or identical.

workgrid
Caught + fixed

A CSP header the spec claimed — absent from the actual config

The security spec said a Content-Security-Policy was set; nginx.conf only had HSTS, nosniff and frame-deny. Fixed: the real CSP (default-src 'self'; object-src 'none'; frame-ancestors 'none'; …) was added.

smartcoder
Caught + fixed

An auth rate-limiter defined, commented "used on auth routes" — never wired

A stricter 10/min limiter existed and its comment claimed it guarded the auth routes, but no decorator applied it, so login ran under the lenient global 100/min. Fixed by actually applying it.

05Injection, XSS & SSRF

Verified by grep and by adversarial test suites, not by hoping.

Parameterized queries throughout; no string-built SQL

Every user-value query uses bound parameters; raw template literals interpolate only compile-time constants (e.g. migration table names) — verified across all eight builds. dangerouslySetInnerHTML is lint-forbidden and CSP is default-src 'self'.

bookflow+ all 8
Verified

SSRF: deny-list + IP pinning + redirect re-validation

Outbound fetches resolve the target, deny cloud-metadata and RFC-1918 ranges, normalize IPv4-mapped IPv6, and re-validate on redirect — with a unit deny-matrix and an integration SSRF_BLOCKED test. Outbound webhooks go through assertPublicHttpsUrl, https-only and re-resolved.

Verified

ReDoS-safe matching on length-capped input

Detectors use a linear-time matcher on 32 KB-capped input; a fixture test asserts under 200 ms against a 50,000-char adversarial string.

06Config, concurrency & correctness

The bugs that don't show up until production load — race conditions, proxy misconfig, and the gate that re-reads the build for logic errors.

Race conditions closed in the database, not the app

No-double-booking is a Postgres EXCLUDE constraint; no-oversell is an atomic decrement plus CHECK(stock >= 0) — both proven race-proof by concurrent tests, instead of a check-then-act the app could lose under load.

bookflowember
Verified

Proxy-blind rate limiting collapsed every client into one bucket

Behind nginx, req.ip wasn't trusted, so the throttler and audit log saw one IP for everyone — defeating per-client limits. Fixed with trust proxy = 1. (Separately: the web tier's default 16 KB header limit returned 431 on large cookies, making a storefront unreachable — caught and sized to match the API tier.)

Webhooks: HMAC over the raw body, before any DB read

Inbound webhooks verify an HMAC over the raw payload with timingSafeEqual before touching the database, inside a replay/staleness window, against an idempotency ledger — tamper, replay, missing-header and bad-signature all rejected.

Honest by design.

The most important thing on this page isn't any single catch — it's that a ⚠ detail never quietly rolls up into a ✓ summary. Every Accepted item carries a rationale and a tracking note. The one critical was surfaced to a human even though it was auto-fixed. And where a control was claimed but missing, the gate would rather escalate it for a decision than fabricate a security primitive to make the checkmark green.

In places the adversarial audit was stricter than the build's own self-assessment — which is the whole point. That's the difference between a build that looks done and one you can actually deploy.
